# Refactor: Backend Shared Utilities, N+1 Queries, and Admin Frontend

## Objective

Eliminate code duplication in the API server backend, fix N+1 query patterns, split the monolithic Admin page into separate components, and replace raw `fetch()` calls in Admin with generated React Query hooks by adding the missing endpoints to the OpenAPI spec and regenerating the client.

## Context & Rationale

### Current Problems

1. **`enrichMatch` duplicated 3 times** — identical logic in `matches.ts`, `bets.ts`, `rankings.ts` each performing per-row DB lookups
2. **`getBettingDeadline` duplicated 2 times** — in `matches.ts` and `bets.ts`
3. **N+1 queries** — `enrichMatch` fires 1-3 queries per match; user lookups in `bets.ts:118-121` fetch one user at a time
4. **Admin.tsx is 513 lines** with 6 tab components in one file
5. **Raw `fetch()` in Admin** — `ScheduleTab` and `ScoringConfigTab` bypass generated hooks because `scoring-config` and `import-schedule` are missing from the OpenAPI spec; they also import a non-existent `@/lib/api` module

### Key Constraint

The `lib/api-client-react/src/generated/` and `lib/api-zod/src/generated/` directories are **auto-generated by Orval** from `lib/api-spec/openapi.yaml`. Any fix to item 4 requires adding the missing endpoints to the OpenAPI spec first, then running `pnpm --filter @workspace/api-spec run codegen` to regenerate the hooks.

## Potential Risks and Mitigations

1. **Orval codegen may produce unexpected output for new endpoints**
   Mitigation: Review the generated hooks after running codegen. The existing endpoints provide a clear pattern for how Orval structures its output. Match the OpenAPI schema shapes to the actual server response shapes precisely.

2. **Batch query refactor may change response shape subtly**
   Mitigation: The `enrichMatch` function's return shape must remain identical. The refactor only changes *how* data is fetched (batch vs. per-row), not *what* is returned. Verify by comparing the JSON response of GET `/api/matches` before and after.

3. **Admin tab extraction may break imports or state sharing**
   Mitigation: The current tabs are already self-contained functions with no shared state between them. The only shared piece is `matchDisplayName` which can be co-located or extracted. The parent `Admin.tsx` only passes the active tab ID.

4. **`credentials: "include"` must be preserved in generated hooks**
   Mitigation: The `customFetch` at `lib/api-client-react/src/custom-fetch.ts` does NOT set `credentials: "include"` by default. The current raw fetch calls in Admin explicitly set this. After switching to generated hooks, verify that the `customFetch` configuration or the Orval-generated code passes `credentials: "include"` for session-based auth. If not, this needs to be added to `customFetch` as a default for all requests.

## Alternative Approaches

1. **For N+1 queries:** Instead of lookup maps, Drizzle supports relational queries (`db.query.matchesTable.findMany({ with: { homeTeam: true, awayTeam: true, group: true } })`). This would be more idiomatic but requires configuring Drizzle relations in the schema, which is a larger change.

2. **For Admin splitting:** Instead of one file per tab, an alternative is a single `admin/` barrel file with co-located tabs. However, individual files are more conventional and provide better tree-shaking potential.

3. **For replacing raw fetch:** Instead of adding to OpenAPI spec, a simpler approach would be to create manual hooks in a non-generated file. However, this defeats the contract-first architecture the project already follows.
